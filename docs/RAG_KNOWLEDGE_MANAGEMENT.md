# 企业知识库与 RAG 运维说明

## 已实现的闭环

知识库文档采用“逻辑文档 + 不可变版本”模型。上传新版本时，线上仍继续检索当前版本；后台完成解析、分块和向量化后，数据库事务一次性写入新分块并切换 `active_version_id`。任何中间失败都不会暴露半成品索引。

版本可以在知识库页面查看和回滚。回滚只切换活动版本指针，不重新计算向量，因此恢复速度快；历史版本的分块仍保留，只有活动版本参与正常检索。

索引任务具有幂等键、领取状态、心跳、重试和过期任务恢复。服务异常重启后，超过 `RAG_JOB_STALE_MS` 的运行中任务会重新排队，避免文档永久停在“处理中”。

检索默认采用混合模式：向量召回与 PostgreSQL 全文/相似词召回分别取候选，再使用 RRF 融合。检索调试台会展示两路候选数、排名、融合得分、文档版本和证据正文。纯向量模式保留用于基线对比。

评测支持同一黄金集分别运行“纯向量”和“混合检索”，并在最近运行中比较 Recall@5、MRR@5、Groundedness 和回答正确率。项目内置 8 组、共 24 条固定黄金问题，证据源位于 `apps/api/test/fixtures/rag/flowmind-product-handbook.md`。

## 数据库升级

按顺序执行 PostgreSQL migration：

1. `003_enterprise_knowledge_versions.sql`：文档版本、活动索引指针、混合检索索引、评测模式和调试数据。
2. `004_rag_index_version_compat.sql`：兼容曾应用过开发中版本迁移的数据库，并补齐必要索引。
3. `005_rag_job_recovery.sql`：任务心跳、尝试次数和过期任务恢复索引。

迁移会把已有文档回填为 V1，不要求重新上传。

## 关键配置

- `RAG_EMBEDDING_PROVIDER`：`openai`、`ollama` 或仅用于开发/测试的 `local`。
- `RAG_VECTOR_TOP_K` / `RAG_KEYWORD_TOP_K`：两路初始候选数量。
- `RAG_FINAL_TOP_K`：融合后返回数量。
- `RAG_RRF_K`：RRF 排名平滑参数。
- `RAG_MIN_SCORE`：最低向量相似度阈值。
- `RAG_JOB_STALE_MS`：索引任务心跳过期时间，生产建议不低于 5 分钟。

生产环境不建议使用 `local` 哈希向量；它用于离线开发和确定性测试，不具备语义模型的召回质量。

## 发布与回退检查

发布前至少执行 API 单测、API/Web/Shared 类型检查，并对一组黄金集分别运行纯向量和混合检索。若新版本指标下降，可立即在文档版本历史中回滚；若检索策略下降，可切回 `vector` 基线运行并对照调试台候选。

角色、权限、审批流和字段级数据访问控制不在本轮范围内，后续可以在现有组织隔离基础上补充，不影响版本与索引模型。
